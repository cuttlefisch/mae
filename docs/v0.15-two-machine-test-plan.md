# v0.15 Two-Machine Test Plan — Collaborative E2E KB Sharing over the mTLS Hub

> **Audience:** two testers ("**alice**" = KB owner + hub daemon host, "**bob**" = member/joiner),
> one per **real machine**, validating the v0.15 release end-to-end across a real network.
> **Scope:** the **mTLS hub** transport (`[collab.auth] mode = "key"`). The P2P **mesh** is
> **out of scope for v0.15** (deferred to v0.16, #263) — do not test `kb-share-p2p`/`kb-join-p2p` here.
>
> This refactors the v0.14 drafts (`collab-testing-plan.md`, `collab-test-notes-*.md`,
> `collab-kb-sync-testing-lessons.md`) into a current, canonical-command plan. Record results in the
> per-machine note files **`docs/v0.15-test-notes-alice.md`** and **`docs/v0.15-test-notes-bob.md`**.
>
> Companion docs: [`DAEMON_ADMIN.md`](DAEMON_ADMIN.md) (operator runbook),
> [`E2E_USER_GUIDE.md`](E2E_USER_GUIDE.md) (the encryption/rotation/recovery workflows),
> and `:help concept:kb-sharing` (in-editor).

## Part 0 — Working protocol (read first, both testers)

### Topology
```
  alice-editor ──unix── alice-daemon ════ TCP + mTLS (key mode) ════ bob-editor
   (owner)              (HUB, bind 0.0.0.0:9473)                     (member, connects in)
```
Only **one** daemon runs (on alice's machine). Bob runs **no daemon** — his editor connects
directly to alice's daemon over mTLS. Both hold a **durable local KB replica**.

### Coordination board (fill in before you start — put in both note files)
| Role | Machine / OS | LAN IP | Build commit | Editor identity fp | Status |
|------|--------------|--------|--------------|--------------------|--------|
| **alice** (owner + hub) | | | | `SHA256:…` | |
| **bob** (member) | | `n/a` (connects to alice) | | `SHA256:…` | |
| **daemon** (on alice) | | `<alice-ip>:9473` | | `SHA256:…` | |

### Conventions
- **Probe slugs** — every test edit carries an UPPERCASE, no-space marker in a node **title/body**
  so it's greppable across machines: `[ALICE-H4]`, `[BOB-H4]`; for concurrent pairs `[A-H11]`/`[B-H11]`.
- **Daemon-log discipline** (alice) — run the daemon with verbose logs teed to a file, and drop a
  timestamp marker before each scenario:
  ```bash
  RUST_LOG="mae_daemon=debug,mae_sync=debug,warn" mae-daemon 2>&1 | tee ~/mae-daemon-live.log
  # before each scenario:
  echo "===== H4 $(date -u +%FT%TZ) =====" >> ~/mae-daemon-live.log
  # then grep from the marker:
  grep -nE "kb/share|kb/join|kb/approve|kb/node_update|DENIED|mTLS|reason=role" ~/mae-daemon-live.log
  ```
- **Reproducibility** — before every run: `make verify-binary` (fail if a stale binary is running);
  pin the commit with `git rev-parse --short HEAD`; to start from a clean slate,
  `scripts/reset-collab-state.sh` (timestamped backup of `collab/` + `kb/`). For deliberate offline
  windows, set `collab_auto_connect` to `false` so the editor doesn't silently reconnect.
- **One row per action** in your note file: `Scenario | My action | Expected | Actual | Status | Evidence`.

### Command-name note
This plan uses the **canonical** names (`:kb-add-member`, `:kb-remove-member`, `:kb-set-policy`,
`:kb-approve`). The legacy spellings (`:kb-member-add`, `:kb-policy`) still work as aliases, but
prefer the canonical ones (they match `:help`, the Scheme prims, and the MCP tools).

## Part 1 — Setup (both machines)

### 1.1 Build (both machines, same commit)
```bash
git fetch && git checkout main && git pull
git rev-parse --short HEAD          # → record on the coordination board
make doctor                         # GUI deps: clang, fontconfig, freetype
make build                          # GUI editor → target/release/mae (+ mae-mcp-shim)
make build-daemon                   # daemon     → daemon/target/release/mae-daemon
make verify-binary                  # no stale running binary
make install                        # → ~/.local/bin/mae, ~/.local/bin/mae-mcp-shim
make install-daemon                 # → ~/.local/bin/mae-daemon   (alice only strictly needs this)
mae --version && mae-daemon --version
```
> GUI auto-falls-back to a terminal UI over SSH / a tty. If a machine is headless and you want a
> lighter build, `make build-tui` works too — the commands below are identical.

### 1.2 Identities + out-of-band key authorization (KEY / mTLS mode)
Membership and auth key on your **Ed25519 fingerprint**, never a username. The daemon must
**authorize each editor's key out-of-band** before it will accept the mTLS connection.

**alice (daemon host):**
```bash
mae-daemon identity                 # → the DAEMON's fingerprint + `mae-ed25519 <b64> …` line
mae --collab-identity               # → alice's EDITOR identity + the exact `mae-daemon authorize …` line
mae-daemon authorize mae-ed25519 <alice-editor-b64> alice
```
**bob (his machine):**
```bash
mae --collab-identity               # → bob's EDITOR identity line; send the `mae-ed25519 <b64>` to alice OOB
```
**alice** authorizes bob, then confirms both:
```bash
mae-daemon authorize mae-ed25519 <bob-editor-b64> bob
mae-daemon authorized               # → lists alice + bob with their SHA256 fingerprints
```
> **Record bob's fingerprint** on the board — you need it for `:kb-approve` / `:kb-add-member` / `:kb-remove-member`.

### 1.3 Daemon config (alice) — `~/.config/mae/daemon.toml`
```toml
[collab]
bind = "0.0.0.0:9473"        # reachable from bob's machine (was 127.0.0.1 — change it)

[collab.auth]
mode = "key"                 # trusted-peer Ed25519 mTLS
tls  = true
```
Start it (foreground, logged) and open the firewall:
```bash
RUST_LOG="mae_daemon=debug,mae_sync=debug,warn" mae-daemon 2>&1 | tee ~/mae-daemon-live.log &
sudo firewall-cmd --add-port=9473/tcp   # or: sudo ufw allow 9473/tcp
ss -tlnp | grep 9473                     # confirm LISTEN on 0.0.0.0:9473
mae-daemon doctor                        # collab port, TLS, authorized count, storage health
```

### 1.4 Editor collab config (both machines)
Use the one-command setup, then verify `init.scm`:
```bash
# alice:
mae setup-collab --server 127.0.0.1:9473
# bob:
mae setup-collab --server <alice-ip>:9473
```
It writes these to `~/.config/mae/init.scm` (add/verify by hand if needed):
```scheme
(set-option! "collab-auth-mode" "key")
(set-option! "collab-server-address" "…")            ; alice: 127.0.0.1:9473 · bob: <alice-ip>:9473
(set-option! "collab-auto-connect" "true")
(set-option! "collab-host-key-policy" "accept-new")  ; NOTE: "prompt" is blocked by issue #66
```

## Part 2 — Scenario matrix (H1–H13)

Run in order. Each scenario lists **alice** and **bob** actions and the **oracle** (what proves it
passed). Grep the daemon log from your `===== Hn … =====` marker.

### H1 — Connect + mTLS auth
- **alice:** launch `mae`; `:collab-connect`; `:collab-status`.
- **bob:** launch `mae`; `:collab-connect`; `:collab-status`; `:collab-doctor`.
- **Oracle:** daemon log shows `mTLS client authenticated` once per peer (alice + bob, distinct
  fingerprints). Both editors' `:collab-status` = **connected**; `:collab-doctor` reports the peer
  reachable, TLS ok.

### H2 — Buffer convergence (text CRDT, pre-KB sanity)
- **alice:** open a scratch file, `:collab-share`.
- **bob:** `:collab-list` → see the shared doc; `:collab-join <name>`; type into it.
- **alice:** type into it simultaneously.
- **Oracle:** the buffer text converges **byte-identical** on both editors; both `:collab-status`
  show `synced-docs: 1`.

### H3 — KB share → invite → pending → approve → join
- **alice:** `:kb-register collabtest <repo>/tests/fixtures/kb/collabtest`; `:kb-share collabtest`.
  - daemon: `kb/share: complete` + `kb/share: owner …` (owner bound to alice's fp).
- **bob:** `:kb-join collabtest` → status "pending owner approval".
  - daemon: `kb/join: pending … principal=<bob-fp>`.
- **alice:** `:kb-pending collabtest` (see bob's fp) → `:kb-approve collabtest <bob-fp> editor`
  (or `SPC C K m` → *KB Sharing* → bob's pending row → `a`).
  - daemon: `kb/approve_member: complete`.
- **bob:** `:kb-join collabtest` again → "Joined (N nodes)".
  - daemon: `kb/join: complete`.
- **Oracle:** the log sequence `kb/share: complete → kb/join: pending → kb/approve_member: complete
  → kb/join: complete`; bob sees the 3+ fixture nodes (`collabtest:overview/alpha/beta`);
  `:kb-sharing-status` on both shows bob = editor.

### H4 — Bidirectional KB CRDT sync
- **alice:** `:kb-update collabtest:alpha [ALICE-H4]`.
- **bob:** open `collabtest:alpha` → title/body shows `[ALICE-H4]` within ~1–2 s.
- **bob:** `:kb-update collabtest:beta [BOB-H4]`.
- **alice:** `collabtest:beta` shows `[BOB-H4]`.
- **Oracle:** daemon `kb/node_update: received` → `kb/node_update: applied wal_seq=N` for each edit;
  `changed=true` on both directions; `wal_seq` monotonic; exactly one apply per edit
  (`grep "wal_seq=N" ~/mae-daemon-live.log | sort | uniq -c` → 1).

### H5 — Roles / policy (ADR-018)
- **alice:** `:kb-set-policy collabtest invite` (confirm via `:kb-sharing-status`).
- **alice:** `:kb-add-member collabtest <bob-fp> viewer` (demote bob).
- **bob (viewer):** `:kb-update collabtest:alpha [BOB-VIEWER]` → **rejected**.
  - daemon: `kb/node_update … DENIED … reason=role 'viewer' may not Edit`. Alice's `alpha` unchanged.
- **alice:** `:kb-add-member collabtest <bob-fp> editor` (restore) → bob edits, now accepted.
- **bob:** try a management command, e.g. `:kb-add-member collabtest <some-fp> editor` → **owner-only DENIED**.
- **Oracle:** viewer write denied at the daemon (no `wal_seq`); editor write accepted; non-owner
  management rejected.

### H6 — E2E encryption + the key-blind canary (ADR-037)
> Pick a per-run canary, e.g. `CANARY=CANARY-$(date +%s)`.
- **alice (before enabling):** `:kb-update collabtest:overview PRECANARY-$CANARY` (a plaintext edit
  that reached the key-blind daemon in the clear — reseal-on-enable must purge it).
- **alice:** `:kb-set-encryption collabtest e2e` (read the one-way/limits advisory it prints).
- **alice:** `:kb-update collabtest:alpha SEALED-$CANARY`.
- **Oracle — key-blind (run on alice's machine):**
  ```bash
  DATA=~/.local/share/mae            # daemon data_dir (per daemon.toml)
  grep -r "SEALED-$CANARY"   "$DATA"           # → EMPTY (ciphertext only)
  grep -r "PRECANARY-$CANARY" "$DATA"          # → EMPTY (purged by reseal-on-enable, #171)
  grep "CANARY" ~/mae-daemon-live.log          # → EMPTY (daemon never logs plaintext)
  ```
- **Oracle — members decrypt:** on **both** alice's and bob's machines, `grep -r "SEALED-$CANARY"
  ~/.local/share/mae/kb` → **PRESENT** (both hold the plaintext locally); bob opens
  `collabtest:alpha` and sees `SEALED-$CANARY`.

### H7 — Member removal → content-key rotation (ADR-037 §D3)
- **alice:** `:kb-remove-member collabtest <bob-fp>` (rotates the content key to a new epoch).
- **alice:** `:kb-update collabtest:alpha POSTROT-$CANARY` (authored under the new key).
- **bob stays connected** (absorbs the ciphertext broadcast he can't open).
- **Oracle — forward secrecy:** on bob's machine `grep -r "POSTROT-$CANARY" ~/.local/share/mae`
  → **EMPTY** (stranded on the old key). **History retention:** `grep -r "SEALED-$CANARY"
  ~/.local/share/mae/kb` on bob → still **PRESENT**. **Key-blind:** `POSTROT` absent from alice's
  daemon data_dir. **Non-vacuity:** alice `grep -r "POSTROT-$CANARY" ~/.local/share/mae/kb` → PRESENT.

### H8 — Identity rotation (ADR-040) + owner self-lockout regression
- **alice:** `:collab-rotate-identity` (reads: authorize the new key, then reconnect).
- **alice (shell):** read the new key — `mae --collab-identity` — then on the daemon host
  `mae-daemon authorize mae-ed25519 <new-alice-b64> alice-rot`.
- **alice:** `:collab-disconnect` → `:collab-connect`.
  - daemon: `mTLS client authenticated` under the **new** fingerprint.
- **★ Regression check (the #266 fix):** alice `:kb-update collabtest:alpha ROTID-$CANARY` **and**
  `:kb-add-member collabtest <carol-fp> viewer` (any owner-only op) → **both accepted** (owner is NOT
  locked out of her own KB after rotate+reconnect). Then undo the test member.
- **bob (still a member):** `collabtest:alpha` converges to `ROTID-$CANARY` (new key = valid owner).
- **Oracle:** new-key mTLS auth in the log; alice can edit + manage post-rotation; bob converges;
  `ROTID` absent from the daemon store (still sealed).

### H9 — Recovery key (ADR-040 §Recovery)
- **bob:** `:collab-register-recovery-key` → saves `~/.local/share/mae/collab/recovery/id_ed25519`.
  Confirm it exists (`ls -l`), then **move it offline** (the guide's instruction).
- **Simulate loss + recover** (a fresh primary — a third machine "carol", or bob after
  `reset-collab-state.sh` keeping only the offline recovery dir):
  - generate a fresh primary (`mae --collab-identity`), have alice `mae-daemon authorize` it;
  - `:collab-recover-identity <recovery-dir> <bob-old-fp>`;
  - `:collab-connect`; `:kb-join collabtest`.
- **Oracle:** daemon accepts the recovery-signed rebind (log: "accepted a member self-service
  identity op"); the fresh primary inherits bob's seat and **decrypts** existing content
  (`grep -r "SEALED-$CANARY" ~/.local/share/mae/kb` → PRESENT on the recovered machine).

### H10 — Offline-merge
- **bob:** `:collab-disconnect`; `:kb-update collabtest:beta [BOB-OFFLINE]`.
- **alice (while bob offline):** `:kb-update collabtest:overview [ALICE-WHILE-OFFLINE]`.
- **bob:** `:collab-connect`.
- **Oracle:** (a) alice's `beta` shows `[BOB-OFFLINE]`; (b) bob's `overview` shows
  `[ALICE-WHILE-OFFLINE]`; (c) no loss — both slugs on both machines; (d) `beta` applied exactly once.

### H11 — Concurrent same-node convergence
- **both:** `:collab-disconnect`.
- **alice:** `:kb-update collabtest:alpha [A-H11]`. **bob:** `:kb-update collabtest:alpha [B-H11]`.
- **both:** `:collab-connect` (order irrelevant).
- **Oracle:** `collabtest:alpha` is **byte-identical** on both machines afterward — converges to **one**
  value carrying both edits' contribution; **not** a split (alice=`[A-H11]` while bob=`[B-H11]`
  would be a lineage/client-id regression — file it).

### H12 — Restart survival (durability)
- **bob editor restart:** `:collab-disconnect`; `:kb-update collabtest:alpha [BOB-DURABLE]`;
  confirm `durable_pending_kb_updates ≥ 1` (via `:introspect` / collaboration section); **quit** the
  editor; relaunch → KB reloads from disk; the pending edit **drains** on reconnect and reaches alice.
- **alice daemon restart:** with a bob edit queued offline, `kill -9` the daemon then restart it.
  - daemon: WAL recovery (`recovering … recovery complete`), membership preserved (no re-approve,
    B-12); bob's during-outage edit converges after reconnect.
- **Oracle:** `durable_pending_kb_updates` survives the editor process restart; WAL replay restores
  the KB; no data loss across either restart; one apply per edit.

### H13 — Security-negative (the attacker's tests)
- **Unauthorized peer:** a third editor whose key is **not** `mae-daemon authorize`d tries
  `:collab-connect` → rejected at the daemon (auth failure in the log; not admitted).
- **Host-key change (MITM defense):** on alice, stop the daemon, delete
  `~/.local/share/mae/collab/id_ed25519` (daemon regenerates a new one), restart. On bob,
  `:collab-connect` → **aborts** with a host-key-changed error until bob re-pins after confirming the
  new key OOB.
- **Confidentiality on the wire:** `sudo tcpdump -A -i any port 9473` during a key-mode session →
  shows TLS records, **not** readable `kb/node_update` JSON-RPC (contrast: `mode = "psk"` is plaintext).
- **Oracle:** unauthorized peer never admitted; changed host-key blocks the connection; tcpdump shows
  no plaintext KB content.

| Surface | Shows |

| `:collab-status` | connection state, server address, synced docs |

| `mae-daemon doctor` | daemon collab port, TLS, authorized count, storage health |

### Key file locations (XDG, both machines)
- `~/.config/mae/daemon.toml` — daemon config (alice)
- `~/.config/mae/init.scm` — editor collab options
- `~/.local/share/mae/collab/id_ed25519(.pub)` — identity seed (editor + daemon each)
- `~/.local/share/mae/collab/recovery/id_ed25519` — offline recovery key (after H9)
- `~/.local/share/mae/collab/authorized_keys` — the daemon's trust store (alice)
- `~/.local/share/mae/kb/` — each peer's local KB replica (the decrypt oracle target)
- `~/.local/share/mae/` — daemon `data_dir` (WAL + snapshots; the key-blind oracle target)

### Troubleshooting
- **Bob can't reach the daemon:** daemon `bind` must be `0.0.0.0:9473` (not `127.0.0.1`); open the
  firewall; check `ss -tlnp | grep 9473` and that `<alice-ip>` is the LAN address.
- **`host-key-changed` on connect:** expected after H13's key deletion — re-pin after OOB confirming,
  or remove the stale entry from `~/.local/share/mae/collab/known_hosts`.
- **Stale binary:** `make verify-binary` before every run; rebuild if it complains.
- **Poisoned state / start fresh:** `scripts/reset-collab-state.sh` (timestamped backup of `collab/`
  + `kb/`); `--list` to see backups.
- **Editor freezes on connect:** ensure `collab-host-key-policy` is `accept-new`, not `prompt`
  (issue #66).

### What this plan does NOT cover (v0.16, tracked)
P2P **mesh** (`kb-share-p2p`/`kb-join-p2p`, daemon↔daemon over iroh) — deferred; a large KB can't pull
over the mesh yet (#263). Use the **hub** for v0.15. The automated CI proof of these flows is
`scripts/collab-{mtls,membership,encrypted}-e2e.sh` (single-host) — this doc is the **real-network**
counterpart.
