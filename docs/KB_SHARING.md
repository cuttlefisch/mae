# Multi-User Encrypted KB Sharing — How It Works in the Wild

> The holistic reference for MAE's collaborative knowledge-base sharing: how **identity**, **authorization**,
> and **encryption** compose into a local-first system where peers share a KB — over a hub or a peer-to-peer
> mesh — and a host that relays it cannot read it. This ties together the deep-dive docs and ADRs:
> `E2E_ENCRYPTION.md` (crypto), `SECURITY_REVIEW.md` (identity + authz audit), ADR-017/018/023/025/026/036/
> 037/038/039. For setup commands see `COLLABORATION.md` + `kb-sharing.md`.

## 1. The shape of it

A **KB** (knowledge base) is a CRDT document graph. **Sharing** one lets several people — and their AI peers
— edit it concurrently and converge, with no central authority required. MAE is local-first: every member
holds a full replica; the network is an optimization, not a dependency.

Three independent layers compose to make sharing safe:

| Layer | Question it answers | Mechanism | Deep dive |
|---|---|---|---|
| **Identity** | *Who are you?* | Ed25519 keypair; the public key **is** the identity (SSH-style, no CA) | `SECURITY_REVIEW.md §3`, ADR-017 |
| **Authorization** | *What may you do?* | A signed, hash-chained **membership op-log** every peer derives identically | `SECURITY_REVIEW.md §4`, ADR-018/023/026 |
| **Encryption** | *Who may read the content?* | A per-KB symmetric **content key**, wrapped to each member through the op-log | `E2E_ENCRYPTION.md`, ADR-037 |

The layers are deliberately separate: identity is the principal, authorization decides membership + roles,
and encryption makes the content private to members so even a hosting daemon stays **key-blind**.

## 2. Identity & trust (who)

Every participant — each editor, each daemon — has a long-lived **Ed25519 identity**. The public key is the
identity; the fingerprint is `SHA256:<base64(sha256(pubkey))>` (full 256-bit, OpenSSH format). There is no
certificate authority. Trust is established exactly as in SSH:

- A daemon trusts peers listed in **`authorized_keys`** (re-read per handshake, so `mae-daemon authorize` /
  `revoke` take effect live). Access control compares the **full key**, never the human label.
- An editor pins a daemon's key on first contact (**`known_hosts`** TOFU); a *changed* key aborts and is not
  re-pinned (OpenSSH `accept-new` semantics). Verify the fingerprint out-of-band, like a Signal safety number.
- On the **mesh**, the join ticket carries the owner's node-id (= its Ed25519 key); the dialer proves the
  live peer matches the ticket before trusting anything — the out-of-band key binding that closes the
  first-contact window.
- Transport is TLS 1.3, but the certificate is only a **carrier** for the Ed25519 key (RFC 7250 raw public
  key). The key, not the cert, is the identity.

See `SECURITY_REVIEW.md §3` for the full trust model + honest weaknesses (at-rest keys, single-key reuse, no
rotation yet — tracked in #158).

## 3. Authorization & membership (what)

A shared KB's access control is a **signed, hash-chained membership operation log** (ADR-026) — *not* a
mutable ACL the host can edit. Each change is an Ed25519-signed `MembershipOp` (admit, remove, role-change,
revoke, set-governance), and **every peer derives the current membership locally and identically** from the
log; the relaying daemon is never trusted to vouch for it.

- **Roles** are hierarchical: **Owner ⊇ Editor ⊇ Viewer** → capabilities **Manage ⊇ Edit ⊇ Read/Join**.
- **The trust anchor** is the genesis owner self-admit, signed by an external root (the owner's key on the
  hub; the ticket node-id on the mesh). A log without a valid anchor yields **zero** members — a relay can't
  self-attest a KB into existence.
- **Delegation** is capability-style (UCAN): an admit can carry `can_invite`, but a grant never exceeds the
  granter's role (attenuation).
- **Removal** uses p2panda strong-removal: concurrent conflicts resolve deterministically (mutual removals
  both apply; a removed member's concurrent actions are invalidated).
- **Governance** is `SingleOwner` (default) or `Quorum{m}` (removing an admin needs *m* co-signatures). E2e
  KBs are `SingleOwner` in v1 (ADR-039 D3).
- **The epoch fence** (ADR-023) stops a member from back-dating pre-grant edits: every write must be authored
  under the member's *current* epoch; stale-epoch ops are rejected.

Enforcement is at the daemon today (the chokepoint `kb_access`), moving to **per-peer** on the mesh (each
peer derives + enforces). See `SECURITY_REVIEW.md §4`; the hardening that unifies role+epoch+blocklist on
every write path is ADR-039 D1 (#157).

## 4. Encryption (who may read)

An owner can mark a KB **`Encryption::E2e`**. Its node *content* is then encrypted under a per-KB symmetric
**content key** (XChaCha20-Poly1305). The key is generated by the owner, **wrapped to each member** via a
sealed box to their identity key, and distributed through the **same signed membership op-log**. A member
recovers its key from the log and decrypts; a hosting/relaying daemon that is not a member carries only
ciphertext and **cannot read the content**. On member removal the owner **rotates** the key, re-wrapping only
to remaining members.

This rides the CRDT payload (not the transport), so it's identical over hub and mesh. Full design,
prior-art positioning, and the honest "what we do **not** protect" (no forward/post-compromise secrecy, O(N)
rekey, metadata leakage) are in `E2E_ENCRYPTION.md`.

## 5. Two transports, one protocol

The same identity + op-log + encrypted op-set flow over either transport — only the wire underneath differs:

- **Hub** (ADR-017): editors connect to a `mae-daemon` over **mTLS** (`key` mode). The daemon hosts the
  collection + node docs and relays updates. For an E2e KB the **owner editor authors** the signed
  membership op-log via the key-blind `kb/collection_op` RPC (ADR-038) — the hosting daemon stores opaque
  owner-signed bytes and never holds the content key.
- **P2P mesh** (ADR-025): daemons connect peer-to-peer over **iroh/QUIC**; the join ticket node-id is the
  trusted-peer key. Membership + content ops are **peer-verified** against the anchored op-log on every relay
  (ADR-026/036) — a relaying daemon can neither forge, mis-attribute, nor (for E2e) read.

A KB's **transport policy** controls where it's reachable (hub-only / mesh-only / both); the owner always
reaches their own KB. The two coexist: one op-log + one content key sync over every transport at once.

## 6. A worked example, end to end

**Alice** owns a KB, enables encryption, and shares it; **Bob** joins and is approved; **Carol** is later
removed. What happens, and who-can-read-what:

1. **Alice enables + shares.** `(kb-set-encryption "research" "e2e")` then `kb-share`. In her network task:
   she generates a content key `k`, persists it `0600`, self-wraps it, and authors a **signed `SetEncryption`
   + genesis self-admit** (the anchor) — shipped via `kb/collection_op`. Her own edits now seal under `k`.
   *The daemon now holds: ciphertext node ops + a signed op-log with `k` wrapped only to Alice. It cannot
   read the content.*
2. **Bob requests access.** `kb-join research`. The daemon records Bob in the collection's **pending** list,
   **including Bob's public key** (from his authenticated session). Alice's editor mirrors the collection and
   surfaces the request in her `*KB Sharing*` buffer.
3. **Alice approves Bob.** `(kb-approve "research" "<bob-fp>" "editor")`. Her network task reads Bob's pubkey
   from the pending record, computes `wrap_to_member(k, bob_pubkey)`, and authors a **signed `Admit`(Bob,
   Editor)** op carrying that `wrapped_key` (dual-writing `member_roles`/epoch). Shipped via
   `kb/collection_op`. *Bob receives the collection, `derive_content_key` recovers `k`, and he reads + edits
   the encrypted content. The daemon still sees only ciphertext + the social graph.*
4. **Carol is removed.** `(kb-remove-member "research" "<carol-fp>")`. Alice generates a **fresh** key `k'`,
   authors a `Remove`(Carol) op + a re-wrap of `k'` to *every remaining* member (Alice, Bob), and persists
   `k'`. *Carol has no wrap of `k'` → she keeps only `k` → she can read history she already had, but **not**
   content authored after the rotation.* (Forward exclusion, not forward secrecy — see `E2E_ENCRYPTION.md §7`.)

At every step the **daemon is key-blind**; trust flows from Alice's anchored signature, not from the host.

## 7. Managing a shared KB

Both the human and the AI peer drive the same lifecycle through the same introspection snapshot:

- **Human:** the magit-style **`*KB Sharing*`** buffer (`SPC C K m`) — members, roles, pending requests,
  policy, governance; plus the Scheme primitives `(kb-share)`, `(kb-join)`, `(kb-set-encryption)`,
  `(kb-add-member)`, `(kb-approve)`, `(kb-remove-member)`, `(kb-set-policy)`, `(kb-sharing-status)`.
- **AI peer:** the MCP tools `kb_sharing_status` (call before managing), `kb_share`, `kb_join`,
  `kb_add_member`, `kb_approve`, `kb_remove_member`, `kb_set_policy`, `kb_leave`.

Owner-only actions (admit/remove/policy/governance/encryption) are `Manage`-gated; a non-owner request is
denied at `kb_access`.

## 8. What's protected — and what isn't

**Protected:** content confidentiality from non-members / relays / hosts (E2e); peer-verifiable membership +
content authorship (a host can't forge or mis-attribute); transport confidentiality + mutual peer auth;
removal denies *future* content (after rotation).

**Not protected (read these honestly):** forward / post-compromise secrecy (a leaked/retained key reads
content under it until rotation — `E2E_ENCRYPTION.md §7`); metadata to the host (edit sizes, timing, author
attribution, the membership/social graph, and — today — cleartext node titles in the manifest); a malicious
*member* (members hold the key by definition — membership control is the only defense); at-rest key exposure
+ no key rotation yet (`SECURITY_REVIEW.md §3`, #158). Known authorization hardening in flight: the unified
op-log fence (#157).
